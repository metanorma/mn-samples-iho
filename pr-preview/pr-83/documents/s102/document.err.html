<title>Data Quality</title>
<p id="_7c6004ff-aaa9-a316-7b27-e728a61305c9">Data quality allows users and user systems to assess fitness for use of the provided data. Data quality measures and the associated evaluation are reported as metadata of a data product. This metadata improves interoperability with other data products and provides usage by user groups that the data product was not originally intended for. The secondary users can make assessments of the data product usefulness in their application based on the reported data quality measures.</p>

001703_​generation_and_​display_of_​gridded_​bathymetryHanging paragraph in clause
<clause id="_generation_and_display_of_gridded_bathymetry" obligation="normative">
<title>Generation and display of gridded bathymetry</title>
<p id="_80cde714-1b8d-f49a-8823-08a0af907625">Most modern hydrographic surveys are conducted using high-resolution multibeam sonar systems. While these systems provide a highly detailed depiction of the seafloor, the storage and processing requirements (that is, data management) can be challenging. A typical hydrographic survey can collect upwards of 10 billion depth estimates over a thirty-day collection period.</p>

<p id="_349d6fcd-7999-ad33-8756-3d5becd0712d">Utilization of a gridded data structure eases the data management concerns of the hydrographer, providing the ability to safely decimate the total sum of collected depth estimates into a manageable quantity of representative nodal depths for processing and production. All gridded datasets should be exposed to rigid Quality Assurance/Control procedures to ensure the final gridded dataset accurately represents the real-world environment. Once a dataset passes an established Quality Assurance/Control process, modern chart production software is used to extract candidate nodal depths from the grid for consideration as final charted soundings.</p>

<title>Structure metadata</title>
<p id="_83f8e2e5-e053-f095-c7a7-dffca08aeb8d">Structure metadata is used to describe the structure of an instance of a collection. Since constraints can be different on separate files (for example they could be derived from different legal sources), or security constraints may be different, the constraint information becomes part of the structure metadata. The other structure metadata is the grid representation and the reference system.</p>

<p id="_8faef6c4-a677-3ebc-9d3c-ab160513777a"> <xref target="fig-s102-structure-metadata"/> shows the <strong>S102_StructureMetadataBlock</strong>. The metadata block is generated by the inheritance of attributes from a number of <eref type="inline" bibitemid="iso19115" citeas="ISO 19115:2003"/> metadata classes and from two implementation classes for the horizontal and vertical reference system. This makes the metadata block a simple table.</p>
